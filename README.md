# RGO Monitoring Bot

Telegram-бот мониторинга и аналитики деловой активности Региональных Групп Офисов (РГО).

## Стек технологий

| Слой | Технология |
|------|-----------|
| Telegram Bot | aiogram 3.x (Python) |
| Язык | Python 3.11+ |
| База данных | PostgreSQL 16+ |
| ORM / Миграции | SQLAlchemy (async) + Alembic |
| AI-аналитика | Anthropic Claude API (claude-sonnet) |
| Транскрипция голоса | OpenAI Whisper API |
| Визуализация | matplotlib (PNG) |
| Планировщик | APScheduler 4.x |
| Контейнеризация | Docker + Docker Compose |
| Логирование | loguru |

## Назначение

Бот ведёт постоянный мониторинг деловой активности 7 региональных руководителей групп офисов (РГО) в 7 корпоративных групповых чатах. Бот добавлен в чаты открыто, но работает в режиме «тихого наблюдателя» — не пишет в группы. Аналитические отчёты получает только Начальник управления (НУ) через личные сообщения.

## Роли

| Роль | Кол-во | Описание |
|------|--------|----------|
| Начальник управления (НУ) | 1 | Единственный получатель аналитики. Управляет ботом через ЛС. |
| Региональный менеджер (РГО) | 7 | Знает о боте. Может подписаться на утренние рекомендации. |
| Руководитель офиса (РО) | N | Участник групповых чатов. Объект косвенного анализа. |

## Основные функции

### Ежедневный отчёт (19:00)

Автоматический отчёт из 13 блоков:

1. Сводка дня — статистика сообщений, топ-участники, молчащие РГО
2. Саммари каждого чата — 5–8 тезисов по каждому из 7 чатов (map-reduce через Claude)
3. Нагрузка РГО — интегральный индекс + bar chart
4. Активность по часам — тепловая карта 7 РГО x 24ч
5. Поручения дня — список со статусами
6. Контекстный анализ поручений — качество исполнения, паттерны
7. Пересылки — анализ пересланных сообщений
8. Упоминания НУ — цитаты + оценка уместности
9. Конфликты и решения — разбор конфликтных ситуаций
10. Инструкции РГО → РО — оценка качества (1–5)
11. Готовность к завтрашнему дню — статусы по чатам
12. Благодарности — оценка культуры признания
13. Рейтинг активности — все участники + stacked bar chart

### Стратегия отправки отчёта (Telegram limit 4096 символов)

Отчёт разбивается на **логические секции**, каждая отправляется отдельным сообщением:

| Сообщение | Содержимое | Формат |
|-----------|-----------|--------|
| 1 | Блоки 1–2: сводка + саммари чатов | Текст (HTML) |
| 2 | Блоки 3–4: нагрузка + активность | `send_photo` (PNG) + caption |
| 3 | Блоки 5–6: поручения + контекстный анализ | Текст (HTML) |
| 4 | Блоки 7–9: пересылки, упоминания, конфликты | Текст (HTML) |
| 5 | Блоки 10–13: инструкции, готовность, благодарности, рейтинг | `send_photo` (PNG) + текст |

- Лимит каждой секции: <= 4000 символов (запас для HTML-тегов)
- Если секция превышает лимит — автосплит по `\n\n` (по абзацам)
- Задержка 300ms между сообщениями (защита от Telegram flood control)
- Графики генерируются в `BytesIO` (без записи на диск) и отправляются как `send_photo`
- Форматирование: `parse_mode=HTML` (`<b>`, `<i>`, `<code>`)

### Детекция поручений (без тегов)

Двухуровневый Claude-классификатор:
- **Уровень 1** (каждые 60 мин) — батч-анализ новых сообщений, детекция поручений по контексту
- **Уровень 2** (18:30) — валидация статусов открытых задач

### Рекомендации РГО (08:30)

Персональные рекомендации подписанным РГО: фокус-область, незакрытые задачи, конкретные действия на день.

### Алерты реального времени

| Тип | Триггер | Задержка | Выходные (Сб/Вс) |
|-----|---------|----------|-------------------|
| `admin_mention` | Упоминание имени НУ в чате | Немедленно | Работает |
| `conflict_detected` | Обнаружен конфликт (Claude/эвристика) | <= 5 мин | Работает |
| `silence_rgo` | РГО молчит > 3ч в рабочее время | Каждый час | **ВЫКЛ** |
| `mass_forward` | > 5 пересылок за 60 мин | <= 5 мин | Работает |
| `keyword_hit` | Слово-триггер из списка НУ | Немедленно | Работает |
| `task_overdue` | Просроченная задача | 12:00, 18:00 | **ВЫКЛ** |
| `self_control_low` | Индекс самоконтроля РГО < 40% | 19:00 | **ВЫКЛ** |
| `bot_removed` | Бот удалён из группового чата | Немедленно | Работает |

### Режим выходных дней (Сб/Вс)

- Ежедневный отчёт: **не генерируется** (если не было активности в чатах)
- Утренние рекомендации РГО: **не отправляются**
- Сбор сообщений: продолжает работать (данные не теряются)
- Рабочие дни настраиваются через `WORK_DAYS=1,2,3,4,5` в `.env` (Пн=1 ... Вс=7)

### Rate limiting команд

- "Тяжёлые" команды (`/report_now`, `/week`, `/month`, `/ask`) — не чаще 1 раза в 5 мин
- Кэширование: если отчёт за сегодня сгенерирован < 30 мин назад — отдать кэш из `daily_reports`
- Перед генерацией — сообщение "Генерирую отчёт, подождите ~30 сек..."
- Деструктивные команды (`/set_role`, `/remove_chat`) — inline keyboard с подтверждением

## Модель данных

### Основные таблицы

- **messages** — сырые сообщения (партиционирование по месяцам через `pg_partman`, auto-create на 2 месяца вперёд). Поля: `edit_history JSONB` (история редактирований), `media_group_id VARCHAR` (группировка альбомов)
- **participants** — демографические данные участников (`user_id PK`, `username`, `full_name`, `first_seen_at`, `last_active_at`)
- **participant_chats** — связь участник-чат с ролью per-chat. Составной PK `(user_id, chat_id)`. Поля: `role VARCHAR(16)` (rgo/ro/nu/other), `joined_at`, `last_active_at`
- **tasks** — поручения, детектированные Claude AI. `task_id SERIAL PK`, unique constraint `(source_message_id, task_text_hash)` для дедупликации
- **daily_reports** — отчёты и кеш аналитики. Поле `model_version` для трассировки версии Claude
- **rgo_recommendations** — рекомендации РГО
- **admin_alerts** — алерты реального времени. Поля: `acknowledged_at TIMESTAMPTZ`, inline keyboard "Принято"

### Служебные таблицы

- **monitored_chats** — динамический список чатов под мониторингом (`chat_id PK`, `chat_title`, `rgo_user_id FK`, `is_active`). Управляется через `/add_chat`, `/remove_chat`
- **alert_keywords** — ключевые слова для алерта `keyword_hit` (`id SERIAL PK`, `keyword TEXT`, `is_active`)
- **api_usage** — трекинг расходов на Claude/Whisper API (`provider`, `call_type`, `tokens_in`, `tokens_out`, `estimated_cost_usd`, `timestamp`)
- **scheduler_runs** — идемпотентность scheduled jobs (`job_name PK`, `last_success_at`, `last_error`)

## Предварительная настройка Telegram

Перед запуском бота выполнить обязательные шаги в BotFather:

1. **Отключить Group Privacy**: `/setprivacy` → `Disabled`
   - Без этого бот получает только команды и @mentions, а не все сообщения
   - Это **критически необходимо** для работы мониторинга
2. **Добавить бота как администратора** в каждый из 7 групповых чатов
   - Даёт доступ к удалённым сообщениям и закреплённым сообщениям
3. **Проверка при старте**: бот автоматически проверяет, что получает сообщения из всех мониторируемых чатов. Если чат не отвечает в течение 6 часов — алерт НУ.

## Безопасность

- `ADMIN_TELEGRAM_ID` — единственный авторизованный пользователь (проверка через middleware)
- Все секреты хранятся только в `.env` (добавлен в `.gitignore`)
- PostgreSQL доступен только с localhost
- Логи содержат только метаданные, без текстов сообщений
- Бэкапы: ежедневный `pg_dump` + GPG-шифрование
- Все внешние вызовы — TLS 1.2+

## Обработка ошибок и отказоустойчивость

### Claude API — retry и circuit breaker

- 3 попытки с exponential backoff (2, 4, 8 сек + jitter) при 429/5xx
- Circuit breaker: 5 подряд ошибок → пауза 15 мин + алерт НУ
- Partial failure в map-reduce: если 1–2 из 7 MAP-вызовов упали — генерировать отчёт по доступным чатам, явно указав "Данные по чатам X, Y недоступны"
- Промежуточные MAP-результаты кэшируются в `daily_reports` для retry без повторного вызова успешных чатов

### Валидация ответов Claude

- Все JSON-ответы парсятся через Pydantic-модели (`model_validate_json()`)
- При невалидном JSON — retry с упрощённым промптом, логирование raw response
- Использовать Anthropic Tool Use (structured output) для гарантированного JSON-формата

### Scheduled jobs — идемпотентность

- APScheduler 4.x с persistent SQLAlchemy JobStore
- При старте бота — проверка пропущенных jobs (catch-up): если отчёт за вчера не был сгенерирован — сгенерировать
- Перед генерацией — проверка `daily_reports.report_date` (не дублировать)
- Таблица `scheduler_runs` фиксирует последний успешный запуск каждого job

### Telegram — обработка событий

- **Бот удалён из чата**: обработка `my_chat_member` update → немедленный алерт НУ: "Бот удалён из чата [название]. Мониторинг чата прекращён." Чат помечается как `is_active=false` в `monitored_chats`
- **Редактирование сообщений**: обработка `edited_message` → обновление `text` в `messages`, сохранение предыдущей версии в `edit_history JSONB`
- **Flood control (429 от Telegram)**: обработка `TelegramRetryAfter` с `await asyncio.sleep(retry_after)`
- **Файлы > 20 MB**: Telegram Bot API не позволяет скачивать файлы > 20 MB. Такие голосовые/видео помечаются как "file too large for transcription"

## Контроль расходов AI API

- Каждый вызов Claude/Whisper API логируется в таблицу `api_usage` (provider, call_type, tokens_in, tokens_out, estimated_cost_usd)
- Дневной лимит: `DAILY_AI_BUDGET_USD=5.00` в `.env`
- При превышении 80% лимита — предупреждение НУ
- При превышении 100% — деградация: отключаются non-critical функции (конфликты, благодарности, instructions_eval), остаются core (task detection, daily report)
- Еженедельная сводка расходов включается в отчёт НУ
